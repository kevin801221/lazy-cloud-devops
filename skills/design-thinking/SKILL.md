# 🎨 Design Thinking — 以人為本的問題解決法

> **方法論致謝**：本 skill 的框架源自 [Stanford d.school](https://dschool.stanford.edu/)
> 與 [IDEO](https://designthinking.ideo.com/) 公開的設計思考教材（公共知識）。內容為 lazy-superstack
> 原創撰寫之可操作版本，非複製任何第三方 repo。

設計思考是一套**以人為本、迭代、偏向行動**的問題解決法。它的核心信念：
**先確定在解「對的問題」，再去把問題解對。** 大多數產品失敗不是因為做得不好，
而是因為一開始就在解錯的問題。

這是一個 **flexible** skill：五階段不是死板的瀑布，而是可前後跳躍、反覆迭代的循環。
但每個階段的「產出物」是硬性的——沒有產出，就不算完成那個階段。

---

## 0. 何時啟動（與何時不要）

**啟動**：要做任何**面向使用者**的產品、功能、流程；需求模糊；不確定使用者真正要什麼；
團隊對「要解什麼問題」沒有共識；要做 0→1 的新東西。

**不要過度套用**：純技術重構、後端效能、明確且已驗證的小修改——這些不需要走完整流程。
設計思考是用來對付「不確定性」的，不是儀式。

---

## 1. 五階段總覽

```
  EMPATHIZE  →  DEFINE  →  IDEATE  →  PROTOTYPE  →  TEST
  同理         定義       發想       原型         測試
   │            │          │          │            │
  理解使用者    框定       廣發想再    做最小       拿給真人
  的真實處境    對的問題    收斂        可驗證的      試，學習
                                       東西
            └──────────── 迭代循環，隨時回到任何前一階段 ───────────┘
```

| 階段 | 核心問題 | 主要產出 |
|---|---|---|
| 1. Empathize 同理 | 使用者實際上**怎麼做、怎麼想、卡在哪**？ | 訪談筆記、觀察、Empathy Map |
| 2. Define 定義 | 我們真正該解的是**哪個問題**？ | POV 陳述、How Might We 問題 |
| 3. Ideate 發想 | 有哪些**可能的解法**？ | 大量點子 → 收斂出 2-3 個 |
| 4. Prototype 原型 | 最小能拿來**問問題**的東西長怎樣？ | 紙稿/線框/可點原型 |
| 5. Test 測試 | 使用者**實際**怎麼反應？學到什麼？ | 測試發現、下一輪迭代方向 |

**發散與收斂（雙鑽石）**：Empathize+Define 是第一個鑽石（先發散理解、再收斂出問題）；
Ideate+Prototype+Test 是第二個鑽石（先發散解法、再收斂出驗證過的方案）。
**永遠先發散、後收斂——不要在發散階段就急著評判。**

---

## 2. 階段一：Empathize（同理）

目的：放下你的假設，去理解使用者**真實**的處境、行為、情緒、動機。
你不是使用者；你的直覺常常是錯的。

**方法**
- **使用者訪談**：開放式問題，問「上次你 X 的時候發生什麼事？」而不是「你會不會用 Y？」。
  問過去的具體事件（真實），不問未來的假設（會說謊）。
- **觀察（fly-on-the-wall）**：看使用者實際操作，注意他們**做的**與**說的**之間的落差。
- **沉浸體驗**：自己走一遍使用者的流程，感受痛點。

**鐵律**
- 問「為什麼」至少五層（5 Whys），挖到根本動機。
- 記錄**原話**與**情緒**，不要當場詮釋。
- 找極端使用者（重度/完全不用），他們會放大需求。

**產出：Empathy Map**（見 `reference/templates.md`）——把觀察分成
SAYS（說）/ THINKS（想）/ DOES（做）/ FEELS（感受）四象限。

---

## 3. 階段二：Define（定義）

目的：把同理階段的混亂觀察，**收斂成一個清晰、可行動、以人為本的問題陳述**。
這是整個流程最關鍵、也最常被跳過的一步。

**工具一：POV 陳述（Point of View）**
用固定句型框定問題：

> **[使用者]** 需要 **[需求]**，因為 **[出乎意料的洞察]**。

範例：「**通勤的上班族**需要**在三分鐘內吃到熱食**，因為**他們把午休時間視為唯一的喘息，
而非進食**。」——注意：洞察（insight）要是「出乎意料」的，不是廢話。

**工具二：How Might We（HMW）問題**
把 POV 轉成可發想的問句，打開解法空間：

> 我們可以怎麼樣（How might we）**讓上班族在喘息的同時、自然地吃到熱食**？

HMW 的甜蜜點：**不要太窄**（「怎麼做一台更快的微波爐」=已預設解法），
**也不要太寬**（「怎麼解決飢餓」=無從下手）。

**鐵律**
- 一次只框**一個**核心問題。貪心框三個 = 一個都解不好。
- 問題陳述裡**不能有解法**。出現「app / 平台 / AI」就是框錯了。

---

## 4. 階段三：Ideate（發想）

目的：對著 HMW 問題，**先衝量、後選質**。數量是品質的前提。

**發散（先做，不評判）**
- **Brainstorm**：限時 10 分鐘，目標 30+ 點子。**禁止批評**，鼓勵瘋狂點子，疊加別人的。
- **Brainwrite**：各自先寫再分享，避免大聲的人主導、安靜的人被埋沒。
- **SCAMPER**：用 替代/結合/調整/修改/挪用/消除/重組 七個角度逼出變化。
- **最壞點子**：故意想最糟的解法，常能反向激出好點子。

**收斂（後做）**
- **點點投票（dot voting）**：每人幾票，貼在最有潛力的點子上。
- **2×2 矩陣**：用「影響力 × 可行性」分類，挑右上角。
- 收斂到 **2-3 個**值得做原型的方向，不要只留一個（太早收斂）。

**鐵律**：發散與收斂**分開做**。同時做 = 每個點子剛冒出來就被掐死。

---

## 5. 階段四：Prototype（原型）

目的：用**最小的成本**，做出一個能拿去「問問題」的東西。原型不是產品，是學習工具。

**保真度由低到高（先低後高）**
- **紙稿 / 草圖**：最快，幾分鐘一個，用來測概念與流程。
- **線框（wireframe）**：測資訊架構與佈局。
- **可點原型（clickable）**：測互動與流程順暢度。
- **綠野仙蹤（Wizard of Oz）**：背後用人工假裝是自動，測「值不值得做」。

**鐵律**
- 原型要**剛好夠用來回答你最大的未知**，不多做一分。過度精美的原型 = 浪費 + 捨不得改。
- 一次做**多個**競爭方案，讓使用者比較，而非只給一個讓他禮貌性點頭。
- 「快速、便宜、能丟掉」是原型的三個美德。

> 🦥 與 `ponytail-guard` 呼應：原型階段的極簡，就是不要把還沒驗證的東西做成正式工程。

---

## 6. 階段五：Test（測試）

目的：把原型拿給**真實使用者**，觀察與學習——不是去說服他們你是對的。

**方法**
- 讓使用者**實際操作**原型，邊做邊講出他在想什麼（think-aloud）。
- **你閉嘴**：不要引導、不要解釋、不要替他點。沉默是你最好的工具。
- 記錄：他卡在哪、誤解了什麼、什麼讓他驚喜、什麼讓他皺眉。
- 問開放式：「這裡你預期會發生什麼？」而不是「這樣好用嗎？」

**測試後**
- 整理發現 → 回到對應階段迭代（常常是回 Define：發現問題框錯了）。
- 一輪測 5 個使用者就能抓到約 80% 的主要問題（Nielsen 法則）。

**鐵律**：測試的目的是**證明自己錯**，不是討讚。使用者的困惑永遠是你的問題，不是他的問題。

---

## 7. 角色化指引

| 你的角色 | 最該用力的地方 |
|---|---|
| **設計師 / 工程師** | Empathize 別跳過；Prototype 保持低保真、捨得丟 |
| **創辦人 / PM** | Define 要狠——框錯問題會浪費整個團隊數月；Test 要誠實面對壞消息 |
| **商業 / 主管** | 把「驗證過的學習」當決策依據，而非最大聲的意見（HiPPO） |

設計思考對應 Lean Startup 的 **Build-Measure-Learn**：Prototype=Build、Test=Measure、
回到 Define/Ideate=Learn。兩者是同一個迭代精神。

---

## 8. 流程檢查清單（每個專案跑一遍）

- [ ] Empathize：有沒有跟**真實使用者**談過？還是在會議室裡腦補？
- [ ] Define：問題陳述裡**有沒有偷藏解法**？POV 的洞察夠不夠「出乎意料」？
- [ ] Ideate：發散時**有沒有衝到 20+ 點子**？還是第三個就開始做了？
- [ ] Prototype：原型是不是**剛好夠驗證最大未知**、且捨得丟？
- [ ] Test：有沒有**閉嘴看使用者操作**？發現的問題有沒有回頭迭代？

任何一格打不了勾，你大概在解錯的問題、或在驗證自己的偏見。回上一步。
